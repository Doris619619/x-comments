# 已知限制与安全规则

## 当前限制

- Goal 2 已使用有头 Playwright 完成两次 50 条真实采集和随机十条抽查；无头模式仍会出现“非法访问”，因此当前真实任务必须使用有头正常浏览器。
- 不保证抓取全部商品，只允许后续 Goal 2 的 3 页或最多 50 条有限范围。
- 已实现持久化搜索清单和默认每 10 分钟一次的单词轮流调度；它不是“每 10 分钟抓完所有词”。首批清单应控制在 3 至 5 个词、单轮最多 50 条；N 个持续到期词时，单个词约每 `N × 10` 分钟一轮。每个词由 `next_due_at`、`interval_minutes` 和未完成任务保护共同控制。
- 已实现基于完整成功采集的 `suspected_missing`/`off_shelf` 状态机，但当前搜索范围最多 3 页或 50 条，仍不能把一次或不完整范围的消失视作商品真实售出。详情页明确售出/下架复查尚未接入自动状态机。
- 新增搜索词需要完成至少一轮安全采集后才会在对应首页分类显示商品；当前不会为了快速填满分类而并发或高频补抓。
- 单次采集最长 120 秒；超时或进程重启遗留的 running 任务会明确失败并释放单 worker，后续关键词仍按安全间隔继续调度。
- 运行时已在 Docker Compose 的 PostgreSQL 16 上完成真实迁移、采集、revision 发布、服务重启和 shopping 隔离镜像同步验证。云端同机部署时，Catalog Sync 对宿主机仅绑定 `127.0.0.1`，shopping 的独立同步容器经共享 Docker 私有网络访问 `x-comments-api:8000`；x-comments 部署负责人负责 PostgreSQL 迁移、每日备份保留 7 天（需要时可延长至 14 天）、真实 `.env`、登录态与令牌的密钥管理。外部失败告警须在服务器配置运维群或邮箱的 webhook，未配置前只能记录 systemd 失败日志。代码不会改写含密的 `.env`。
- API 创建的任务先持久化为 `pending`，唯一 scheduler-worker 通过短轮询原子认领，进程内队列
  只是同进程的低延迟提示；worker 重启后未认领任务仍可恢复。没有 Redis/Celery 分布式队列，也不允许
  把 scheduler-worker 扩容为多个爬虫副本。
- 容器依赖宿主机先完成人工登录；纯容器不能自动生成合法登录态。
- 登录态会过期，需要用户在本机重新人工登录。
- 闲鱼页面字段和结构可能变化，离线 fixture 通过不能替代真实抽查。
- 未登录页面可能返回与关键词不匹配的公共推荐，不能作为任务成功证据。
- 商品 API 仅面向受控的服务器端到服务器端读取；当前未配置 CORS，浏览器直连会受同源策略限制。
- 商品 API 的 `image_url`、`image_urls` 和 `location` 可为空或为空数组。详情图库只采集公开图片且最多九张，并共享默认 30 秒的单次采集总预算；预算耗尽或单个详情页超时时保留搜索页封面，不把可选图库失败升级为目录失败。当前真实详情页的图库使用不带 `main` 容器的 `fadeInImg` CSS Modules 类，选择器同时保留旧结构兼容。登录失效或风控仍会立即安全停止，商城必须提供缺图和无地区降级，API 不会伪造数据。
- Catalog Sync 增量不会返回 `item_url`；shopping 只能使用标题、CNY 价格、图片、地区、状态和 revision 创建自己的镜像。游标失效返回 409 时必须进行分页全量重建，不能把 409、401、5xx 或网络错误解释为商品下架。
- 结算核验依赖当前闲鱼详情页的最终商品 URL、明确不可售文案和有限价格选择器；结构变化、
  重定向或价格无法唯一提取时只返回 `unknown`，不会猜测为可售。合并代码不等于已经完成
  真实登录态抽查，部署前仍需在受控环境验证至少一件可售与一件已售商品。
- 当前详情页主价格由客户端延迟渲染，且页面下方同时存在多个推荐商品价格。核验器最多在总预算内
  等待主商品信息区的唯一价格节点，禁止退化为全页模糊价格扫描；若主节点超时、重复或消失，必须
  返回 `unknown` 并停止采购对话。
- Chromium 或详情页可能在实时核验期间提前关闭。系统只在业务结果已经产生后的清理阶段忽略精确的
  `TargetClosedError`；业务核验阶段发生同类异常会返回 `verification_target_closed` 和
  `unknown`，不会重试、放宽账号身份门禁或判断商品可售。
- 同一账号串行锁覆盖唯一 scheduler-worker 进程；API 可以水平扩容，但 scheduler-worker 必须保持
  单副本。增加 worker 数量会绕过账号锁并提高风控风险，因此不在本 POC 的部署范围内。
- `POST /api/v1/items/{item_id}/verify` 必须配置 `XIANYU_API_TOKEN` 并由商城服务器携带；未
  配置时返回 503。当前令牌是 POC 服务间共享密钥，不替代生产级网络隔离、密钥轮换或审计系统。
- `CATALOG_SYNC_TOKEN` 仅支持一个当前有效值。轮换时由运维生成新随机值、先更新 x-comments 和
  shopping 的服务端 secret、滚动重启并验证 401/200；发生泄漏时立即替换并重启两端。令牌不得放入
  `NEXT_PUBLIC_*`、日志、数据库或 Git。
- 采购任务 API 的创建只使用本地 Item 与最新发布 Catalog 快照。彦诗筛选源不再在打开聊天前执行
  商品详情页实时核验；聊天编排会校对本地商品 ID、官方 URL 和 CNY 快照，并在页面上确认商品、
  卖家与买家账号绑定。草稿持久化、单次发送保护和事务 Outbox 已接入。
- v1 与 v2 采购 API 都要求 `PROCUREMENT_SOURCE_ITEM_ALLOWLIST`；v2 还依靠商城逐任务授权，
  两层门禁必须同时成立。两者都不能替代聊天商品、卖家、账号和风控绑定。标题
  疑似夹带客户、联系方式、卡号或支付资料时返回 422，响应不回显命中文本。
- 采购创建 API 只接受商品 ID，不接受调用方 URL；仍须依赖本地目录 URL 的采集正确性。最新发布
  快照只需存在 CNY 价格且与商城一致；`active`、`suspected_missing`、`off_shelf` 等目录状态不会
  阻断任务。这表示信任彦诗选品，不表示实时库存确认；真实可用性由后续卖家对话和人工采购判断。
- `PROCUREMENT_API_TOKEN` 必须独立配置且至少 32 字符；当前只支持一个有效值，不替代生产网络隔离、
  Secret 轮换与来源限制。
- `PROCUREMENT_CHAT_ENABLED` 默认关闭，但当前生产已为草稿验收开启；`PROCUREMENT_AUTO_SEND_ENABLED`
  仍默认并保持关闭。v2 任务还需要任务级可信授权；只有单条 Root Canary 的受控发送窗口可临时开启
  自动发送。正式启用前必须完成受控账号人工验收、
  页面身份二次核验、点击后崩溃对账、限速和紧急停用演练；仅通过离线单元测试不能授权真实发送。
- 真实聊天页已完成一次只读 DOM 标定，确认了入口、输入框、发送按钮、左右消息方向和
  `column-reverse` 顺序；账号绑定使用 `tracknick` SHA-256。该标定没有发送消息，也不能证明未来
  页面结构、账号状态或平台风控不会变化，第一次真实发送仍必须是 Root 单商品 Canary。
- 2026-07-24 的前两次真实发送 Canary 中，`fill + click` 与 `fill + Enter` 都未形成可见消息，
  后续只读核对确认基线后没有本人同文消息。控件诊断确认文字显示、按钮启用且标签为“发 送”，
  因此发送适配改为逐字键盘事件输入后只点击一次语义按钮，并把确认窗口延长至约十秒；结果不确定时
  仍不允许回退 Enter、再次点击或自动重试。
- 当前发送补丁进一步改为可见鼠标移动、按下和松开，并要求匹配草稿的 XHR/fetch 请求或 WebSocket
  出站帧与本人消息回显双重确认。它不会绕过验证码或风控，也不能保证闲鱼未来继续使用可检查的明文
  载荷；若平台改为无法匹配正文的二进制或加密协议，系统会保守转人工而不是降低证据标准。
- 当前商品页聊天入口的动态类名与可见文案会变化；代码只接受客户端延迟渲染后唯一可见且同时携带
  `itemId`/`peerUserId` 的 IM 链接，并继续核对商品、卖家和买家账号。参数结构变化、候选不唯一
  或身份不一致时任务会失败关闭，不能回退到全页任意消息入口或猜测卖家身份。入口可见后允许最多
  约两秒只读等待 `href` 稳定，并以同一次浏览器执行原子读取可见性和身份 URL；仍未稳定时转人工，
  不会重新导航或尝试发送。
- 采购模型不能决定购买、付款、议价、修改地址或确认收货。站外链接、联系方式、付款、客户地址、
  购买承诺、价格变化、卖家/商品不匹配、三轮上限和低置信度均会被确定性策略阻止。
- DeepSeek 草稿调用仅接受严格 JSON；超时、传输、非 200、截断/过滤、包络变化、Schema 偏离、
  伪造消息证据或五类问询外意图都会失败关闭。当前没有自动重试或供应商降级，也不会把供应商错误
  正文写入异常或日志；调用方仍须自行通过 Secret 管理 API Key。
- Outbox 是至少一次回调而不是端到端恰好一次；商城必须按 `event_id` 幂等处理。某任务存在未交付事件
  时 worker 不会继续该任务的外部聊天动作，永久回调失败会让该任务保持阻断并等待人工处理。
- x-comments 只负责核实商品并生成/发送受控询问，不知道虚拟 JPY 测试支付、真实 CNY 人工采购支出，
  也不执行购买、付款、退款或客户履约；这些状态仅由商城管理后台持有。

## 强制停止条件

出现验证码、安全验证、登录失效、登录/验证页跳转、HTTP 403/429、访问频繁、账号异常、操作受限、搜索结果异常归零、连续请求失败或关键结构异常时，任务必须停止并标记 `blocked_by_auth_or_risk_control`，禁止自动重试。

## 明确不实现

验证码识别/绕过、签名破解、访问控制绕过、代理池、多账号轮换、高频并发、手机号采集和精确地址
采集均不在项目范围内。采购聊天未来只允许任务关联的有限对话，不允许批量私聊采集。

`storage_state.json` 是敏感本地文件，已被 Git 忽略；不得提交、上传、打印或写入数据库。

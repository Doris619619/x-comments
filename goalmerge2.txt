# 云端 x-comments 定时采集与 shopping 同步交接目标

## 1. 文档目的与适用范围

本文件定义第二阶段的跨仓库交接目标，覆盖图片任务中的：

```text
3. 完善数据库的定时爬虫
4. shopping 仓库定时访问数据
```

涉及两个仓库：

```text
x-comments             闲鱼公开商品采集、状态判断和对外只读 Catalog API 的唯一所有者
c-shopping-jp-poc      商城展示、购物车和订单侧的本地同步消费者
```

目标部署环境为云服务器。`x-comments` 仍只采集正常可访问的公开搜索结果；不实现验证码绕过、登录绕过、代理池、多账号轮换或访问控制绕过。

本文件不实现购物车、扣款或订单流程本身，只定义它们读取商品可用状态时必须遵守的边界。

---

## 2. 已确认的核心结论

### 2.1 从第一阶段起直接使用 PostgreSQL；shopping 不直接读库

本阶段不再采用 SQLite 作为运行时数据库。开发、测试集成环境和云端部署均以 PostgreSQL 为唯一目标数据库；本地通过 Docker Compose 或受控的开发 PostgreSQL 实例启动。

`x-comments` 独占自己的 PostgreSQL 数据库、schema 和迁移。shopping 保留其既有 MongoDB
交易目录和迁移；shopping **不得**使用 x-comments 的数据库账号、schema、表或只读连接来同步商品。

跨仓库边界固定为：

```text
x-comments PostgreSQL（唯一写入方）
       ↓ HTTPS + 服务间认证
Catalog 同步 API（只读、版本化）
       ↓
shopping 服务端定时拉取并写入自己的商品镜像
```

理由：

- 两个服务的部署、迁移和回滚不会互相破坏；
- shopping 不依赖采集库的内部表结构；
- 两个服务不需要共享数据库文件、schema 或表级写锁；
- 可以为 shopping 只暴露最小字段和稳定版本契约；
- 采集服务故障时，shopping 仍可用上一次已完成同步的商品镜像。

PostgreSQL 是从本阶段起的唯一运行时数据库选择：

- `x-comments` 使用独立 PostgreSQL 数据库，例如 `x_comments`；
- shopping 保留独立 MongoDB 商品镜像，不做超出本目标范围的全库 PostgreSQL 迁移；
- 两者只通过 HTTPS Catalog Sync API 交接数据；
- 本地开发、CI 集成测试和云端部署均提供 PostgreSQL 服务或连接串；不得将 SQLite 作为兼容分支、共享文件或生产降级方案。

### 2.2 不在整个爬取期间“锁数据库”

Playwright 访问页面、等待响应、解析和去重期间，不能持有数据库写锁。一次爬取可能持续较久，遇到超时或风控还可能失败；若全程锁库，会使读取被阻塞并提高 PostgreSQL 锁冲突和连接耗尽风险。

正确的发布模型：

```text
开始采集
  → 创建 crawl run，状态 running（短事务）
  → 浏览器采集、解析、校验（不持有 DB 事务）
  → 结果完整成功后，短事务写入商品、关联和状态变化
  → 在同一短事务中创建 published catalog revision
  → shopping 只能读取已 published 的 revision
```

因此 shopping 在写入发生时只会读到：

- 上一个完整发布版本；或
- 新的完整发布版本。

不会读到半批商品或“数据库锁住”的特殊内容。数据库事务与发布版本本身就是一致性保证，不需要额外设计一个全库锁状态给 shopping 读取。

### 2.3 搜索结果单次未出现，不等于商品已下架

闲鱼搜索排序、可见范围、登录状态和风控都会影响结果。特别是当前采集范围有限时，商品从前 50 条结果中消失，不等于商品真实售出或下架。

因此禁止使用以下规则：

```text
本轮搜索没看到商品 → 立即下架
```

默认状态规则为：

```text
本轮采集失败 / 风控 / 结果不完整
  → 仅更新本轮确实看到的商品；绝不改变未看到商品的可用状态

本轮完整成功，但商品未出现
  → missing_count + 1，状态为 suspected_missing，不向主页推送

连续 2 次完整成功均未出现
  → off_shelf，商品从公开首页隐藏

商品详情页明确显示已售出或已下架（后续合规复查功能）
  → 可直接标记 sold / off_shelf
```

连续缺失次数默认取 `2`，必须可配置并在文档中说明。若某关键词的搜索只能采集有限窗口，不能仅凭缺失自动把商品标为 `off_shelf`；此时只能标记为 `suspected_missing`，或增加合规的、低频详情页复查后再确认。

一个商品可能命中多个关键词。商品只有在**所有有效关键词关联都不再 active**时，才从全局公开目录隐藏；任一关联仍 active，就不能全局下架。

## 3. 云端目标架构

```text
                    ┌────────────────────────────────────┐
                    │ x-comments                           │
                    │                                    │
登录态（机密挂载） ──→ 单实例 scheduler / worker             │
                    │        ↓ Playwright                 │
                    │   x-comments PostgreSQL（唯一写入方） │
                    │        ↓ 原子发布 revision           │
                    │   只读 Catalog Sync API             │
                    └───────────────┬────────────────────┘
                                    │ HTTPS + 服务认证
                                    ▼
                    ┌────────────────────────────────────┐
                    │ shopping                             │
                    │  每 10 分钟服务端同步任务             │
                    │        ↓                             │
                    │  shopping MongoDB + 同步游标          │
                    │        ↓                             │
                    │  首页仅 active；购物车保留下架条目     │
                    └────────────────────────────────────┘
```

### 3.1 x-comments 云端职责

1. 保存可编辑、持久化的关键词采集清单；
2. 按安全节奏创建采集任务，并保证同一清单项不会并发执行；
3. 记录每次采集是否“完整成功且可用于状态比较”；
4. 维护商品、商品-关键词关联、可用状态、状态变化和发布版本；
5. 仅在完整成功后原子发布新的 Catalog revision；
6. 提供稳定、只读、版本化的同步 API；
7. 安全停止登录失效、验证码、403/429、异常页面等风险场景；
8. 输出最后一次成功采集、最后一次发布时间和失败原因，供运维监控。

### 3.2 shopping 云端职责

1. 仅从服务器端调用 x-comments API；浏览器不能拿到服务地址密钥或同步令牌；
2. 每 10 分钟拉取自上次游标后的变更，幂等写入自己的商品镜像；
3. 不直接修改 x-comments 商品状态，也不反向写入采集数据库；
4. 首页、搜索页和推荐位只显示本地镜像中 `active` 的商品；
5. 已加入购物车的商品若变为 `suspected_missing`、`sold` 或 `off_shelf`，保留购物车记录并显示对应不可售提示；
6. 加购和提交订单前均以 shopping 的最新同步状态再次校验；不可售商品不得进入扣款/下单；
7. x-comments 暂时不可达时保留上次同步结果，并在运维日志中记录同步失败；不能把网络失败误判为所有商品下架。

## 4. 关键词清单与“每 10 分钟”调度规则

### 4.1 清单模型

关键词必须存储在数据库清单中，而不是写死在代码或环境变量中。每项至少包含：

```text
id
category
keyword
is_enabled
interval_minutes
last_scheduled_at
last_completed_at
next_due_at
last_run_status
note
```

`keyword` 全局唯一；禁用条目不会被调度；编辑清单不删除已采集商品历史。

### 4.2 默认调度语义

默认采用安全、可审计的策略：

```text
x-comments 启动后立即调度一个到期清单项；
之后每 10 分钟调度一个到期清单项；
同一时刻只运行一个 Playwright 采集任务；
同一关键词未完成时不得重复入队；
若上次任务未完成，则本周期不并发启动第二个浏览器任务。
```

这表示“系统每 10 分钟进行一次采集尝试”，而非“清单内每个关键词每 10 分钟都爬一次”。清单有 `N` 个持续到期词时，一个词的最短实际轮询周期约为 `N × 10` 分钟，加上单次爬取耗时。

若业务要求“每个关键词都必须每 10 分钟更新”，需要先缩小清单到可在 10 分钟内安全串行完成的数量；不能为了满足频率启用浏览器并发、账号轮换或代理池。该需求须作为单独容量验证项，不能默认实现。

### 4.3 云端单调度器保证

云端扩容 API 副本时，不能让每个副本都启动调度器，否则会重复爬取。第一版采用以下任一明确方式，实施时二选一并记录：

- **推荐**：单独部署一个 `scheduler-worker` 副本，API 服务副本不运行 scheduler；
- 或：在 PostgreSQL 中使用带过期时间的 scheduler lease，只有持有 lease 的实例能调度。

无论选择哪种方式，必须有自动化测试或可重复验证，证明两个实例同时启动时同一清单项不会产生两个并发任务。

## 5. 数据模型与原子发布要求

### 5.1 必须新增或扩展的数据

现有 `items`、`keywords`、`item_keywords`、`crawl_jobs`、`catalog_keywords` 保留。为支持可靠同步，需要新增/扩展以下概念：

| 概念 | 最小字段 | 用途 |
| --- | --- | --- |
| `crawl_runs` | `run_id`、`catalog_keyword_id`、`status`、`started_at`、`finished_at`、`is_comparable`、`error_message` | 区分运行成功、失败和是否可用于“缺失”判断 |
| 商品-清单状态 | `item_id`、`catalog_keyword_id`、`availability`、`missing_count`、`last_seen_at`、`last_checked_run_id`、`status_changed_at` | 不能只用商品全局 `last_seen_at` 判断下架 |
| `catalog_revisions` | 单调递增 `revision`、`published_at`、`source_run_id`、`status` | 为 shopping 提供一致、可追踪的同步版本 |
| `catalog_changes` | `revision`、`item_id`、`change_type`、`availability`、`occurred_at`、可展示字段快照 | 向 shopping 增量交付上架、更新、下架事件 |

`availability` 至少包括：

```text
active
suspected_missing
sold
off_shelf
unknown
```

公开目录是否展示由服务端统一计算：仅 `active` 可出现在新用户的公开列表中。`suspected_missing` 默认不推送首页，但 shopping 已有购物车仍可查询并展示其状态。

### 5.2 事务边界

一次“可发布”的完整采集必须在一个短数据库事务中完成：

1. 更新本轮看到的商品和商品-清单关联；
2. 对本关键词本轮未看到的旧关联按规则递增 `missing_count`；
3. 计算关联和商品全局可用状态；
4. 写入所有状态变化事件；
5. 创建递增的 `catalog_revisions` 记录；
6. 将相应 `catalog_changes` 全部绑定到该 revision；
7. 提交。

提交失败时整批回滚，不能发布 revision。爬取失败、风控、解析错误或结果不完整时不得执行第 2～6 步，不得发布“空版本”。

## 6. 跨仓库 API 契约

### 6.1 网络和认证

- 优先将两个服务置于同一私有网络/VPC；若 API 必须公网可达，必须启用 HTTPS、来源限制和服务间认证；
- shopping 使用仅服务器端保存的 `X_COMMENTS_SYNC_TOKEN`；不得使用 `NEXT_PUBLIC_*`，不得输出至浏览器或日志；
- x-comments 仅允许只读同步权限，token 不具备创建任务、修改清单或读取登录态的能力；
- `storage_state.json`、Cookie、密码和任何登录态绝不通过 API 返回。

### 6.2 同步端点（v1 契约）

现有 `/api/v1/items` 继续服务于人工查看/普通列表。新增专供服务间同步的端点，不让 shopping 依赖内部表：

```text
GET /api/v1/catalog-sync/revisions/latest
GET /api/v1/catalog-sync/changes?after_revision={integer}&limit={1..500}
GET /api/v1/catalog-sync/items/{item_id}
```

`GET /revisions/latest` 最小响应：

```json
{
  "revision": 42,
  "published_at": "2026-07-16T10:00:00Z",
  "source": "xianyu",
  "status": "published"
}
```

`GET /changes` 必须按 `revision ASC, item_id ASC` 稳定排序，且必须包含：

```json
{
  "from_revision": 40,
  "to_revision": 42,
  "has_more": false,
  "changes": [
    {
      "revision": 41,
      "change_type": "upsert",
      "item_id": "1234567890",
      "availability": "active",
      "title": "商品标题",
      "price": "12.50",
      "currency": "CNY",
      "image_url": "https://...",
      "location": "杭州",
      "last_seen_at": "2026-07-16T09:59:00Z",
      "status_changed_at": "2026-07-16T10:00:00Z"
    },
    {
      "revision": 42,
      "change_type": "availability_changed",
      "item_id": "9876543210",
      "availability": "off_shelf",
      "status_changed_at": "2026-07-16T10:00:00Z"
    }
  ]
}
```

约束：

- 价格以字符串十进制数返回，shopping 不得将其当作 JPY 或进行隐式币种转换；
- `item_id` 是稳定外部标识；
- `availability_changed` 必须保留在增量流中，不能从列表静默消失；
- 当 `after_revision` 太旧且变更历史已清理时，API 返回明确的 `409 full_resync_required`，shopping 必须执行分页全量重建；
- API 超时、401/403、429、5xx、格式错误均不得被 shopping 解释为商品下架；
- API 契约版本、字段增删和弃用规则必须写入两个仓库的文档，并提供契约测试。

### 6.3 shopping 同步算法

```text
每 10 分钟：
  1. 从本地读取 last_applied_revision。
  2. 请求 changes?after_revision=last_applied_revision，直到 has_more=false。
  3. 每一页在 shopping 本地事务中幂等 upsert 商品镜像和状态。
  4. 同一事务中推进 last_applied_revision。
  5. 成功后记录 sync_at；失败时保留旧 revision，下一周期重试。
```

不得在应用内存中只记录游标；重启后必须可恢复。重复拉取同一 revision 不得产生重复商品、重复购物车项或错误状态回退。

## 7. 购物车、主页与订单边界

| 场景 | 首页/搜索 | 已存在购物车 | 下单/扣款 |
| --- | --- | --- | --- |
| `active` | 展示 | 正常展示 | 可进入正常校验 |
| `suspected_missing` | 不推送/隐藏 | 保留并提示“正在确认商品状态” | 禁止提交 |
| `sold` | 隐藏 | 保留并提示“商品已售出” | 禁止提交 |
| `off_shelf` | 隐藏 | 保留并提示“商品已下架” | 禁止提交 |
| x-comments 同步失败 | 使用上次同步结果 | 不批量改状态 | 不以失败为由批量下架 |

购物车记录必须保存 `source=xianyu`、外部 `item_id`、加入时商品标题/价格快照和当前同步状态。不得因商品下架而删除用户购物车记录；删除只能由用户主动操作或明确的数据保留策略执行。

## 8. Goal 队列

任一时刻只允许一个 Goal 为 `in_progress`。状态只能是 `pending`、`in_progress`、`passed`、`blocked`、`failed`。未达到全部验收标准不得进入下一 Goal。

### Goal 0：双仓库契约冻结与云端配置决策

**状态：`in_progress`**

**目标**：冻结关键词调度语义、网络拓扑、服务认证方式、下架阈值、保留期和 shopping 商品镜像模型。

**必须完成**：

1. 确认第 9 节列出的待确认事项；
2. 记录两个仓库的当前 `git status --short`，不得覆盖已有改动；
3. 在两个仓库分别写入同一份 API 契约和环境变量清单；
4. 明确 x-comments PostgreSQL 的连接来源、备份方式和迁移执行者，以及 shopping MongoDB 的既有运维归属；
5. 明确 x-comments 的 worker/scheduler 是否独立部署；
6. 为 API token 规定创建、轮换和撤销方式，且不提交真实值。

**通过标准**：所有决定已落入文档；购物仓库确认能实现同步端点客户端；无直接数据库访问方案残留。

### Goal 1：PostgreSQL 数据基础与单调度器

**状态：`pending`，仅在 Goal 0 通过后启动。**

**目标**：将 x-comments 的运行时数据库直接统一为 PostgreSQL，并使关键词清单和单实例定时调度在本地、CI 和云端均可恢复、无重复。

**验收标准**：

1. 空 PostgreSQL 可通过 Alembic 迁移到 head；Docker Compose、本地运行说明和 CI 集成测试均实际连接 PostgreSQL；
2. API、worker/scheduler 的数据库 URL 均来自环境变量，默认示例和运行文档均为 PostgreSQL URL；
3. 启动两个 scheduler 实例的验证中，每个到期关键词只会创建一个任务；
4. 重启后清单、`next_due_at`、任务状态和最近执行信息不丢失；
5. scheduler 每 10 分钟的行为和“一个到期项”语义有自动化测试；
6. 登录态仅作为云端 secret/受限文件挂载，未进入 Git、数据库、API 或日志；
7. 不保留 SQLite 数据库 URL、SQLite 专用运行分支或 SQLite 容器卷作为本阶段的运行路径；
8. `pytest`、`ruff check .`、`mypy app` 通过，并更新架构、数据模型、配置、运行与限制文档。

### Goal 2：x-comments 原子发布、状态判断与同步 API

**状态：`pending`，仅在 Goal 1 通过后启动。**

**目标**：实现可比较采集、缺失状态机、revision 和只读增量 API。

**验收标准**：

1. 一轮完整成功能在单短事务发布一个 revision；失败/风控/部分结果不发布 revision，也不增加未看到商品的缺失次数；
2. 已看到商品刷新 `last_seen_at` 并重置 `missing_count`；连续两次完整缺失才进入 `off_shelf`；
3. 多关键词商品仅在所有关联均不可用时从公开目录隐藏；
4. 同步 API 具备 token 鉴权、稳定排序、分页和旧游标 `409` 行为；
5. API 不泄露内部 ID、登录态、cookie、任务错误堆栈或数据库连接信息；
6. 单元、仓储、API 契约测试覆盖 active、失败采集、首次缺失、连续缺失、多关键词、重复请求和游标恢复；
7. 同步更新 `docs/architecture.md`、`docs/data-model.md`、`docs/api.md`、`docs/known-limitations.md` 和部署文档。

### Goal 3：shopping 定时同步与商品镜像

**状态：`pending`，仅在 Goal 2 通过后启动。**

**目标**：shopping 每 10 分钟可靠拉取 x-comments 已发布的增量，并将商品状态同步到自己的存储。

**验收标准**：

1. 同步仅从 shopping 服务器端发生，令牌不会进入客户端 bundle；
2. 首次全量同步和后续增量同步均能恢复，并持久化 `last_applied_revision`；
3. 重复执行同一同步批次结果幂等；
4. `upsert`、价格/图片更新、`suspected_missing`、`sold`、`off_shelf` 均准确反映到本地商品镜像；
5. x-comments 超时、401/403、5xx 或格式错误时，不推进游标、不批量下架、不破坏既有商城页面；
6. 针对分页、409 全量重建、重复事件、乱序保护和失败重试编写自动化测试；
7. 更新 shopping 的架构、环境变量和运行文档，且不将 x-comments 数据库地址/凭证作为购物仓库配置。

### Goal 4：shopping 展示与购物车下架行为

**状态：`pending`，仅在 Goal 3 通过后启动。**

**目标**：让公开商品入口与购物车正确反映同步状态，确保下架商品不会继续被推送或进入扣款流程。

**验收标准**：

1. 首页、搜索和推荐只展示 `active` 的闲鱼商品；
2. 已在购物车中的 `suspected_missing`、`sold`、`off_shelf` 商品仍可见，并有清楚的不可售提示；
3. 对上述状态的商品禁用加购、结算和下单；服务端也必须拒绝绕过前端的提交；
4. 购物车保留加入时标题/价格快照，状态变化不会丢失用户记录；
5. 商品重新变为 `active` 后，界面和服务端校验可恢复正常；
6. 断开 x-comments 服务时，商城已有页面和购物车不白屏，且不会错误显示全部下架；
7. 覆盖桌面、移动端以及状态切换的自动化/浏览器验证；通过 shopping 的 lint、类型检查、构建与现有回归测试；
8. 两仓库文档说明“采集结果不是实时库存保证”，不能把不确定状态展示成可购买承诺。

### Goal 5：云端端到端演练、监控与交接

**状态：`pending`，仅在 Goal 4 通过后启动。**

**目标**：在真实云端环境完成一次可审计的采集→发布→同步→展示→下架状态演练。

**验收标准**：

1. 两服务可按文档独立部署、重启和回滚；
2. 记录一次真实成功采集、revision 发布、shopping 同步和前台展示的证据；
3. 用受控测试记录验证一次状态变化不会在中间状态暴露；禁止伪造真实采集成功；
4. 健康检查/监控至少覆盖：最后成功采集时间、最后 published revision、最后成功同步时间、连续失败次数和同步滞后 revision；
5. 设定告警阈值：超过约定时长未成功采集或未同步时告警，但不自动标记商品下架；
6. 最终审查两个仓库：无 Cookie、Token、密码、`storage_state.json`、数据库备份或真实个人信息进入 Git；
7. 在各自 `docs/goals/` 写入实际命令、实际结果、已知限制和最终状态。

## 9. 已确认的部署与业务决策（2026-07-17）

1. **调度语义**：每 10 分钟只处理一个到期关键词；N 个持续到期词时，单词约每 `N × 10` 分钟一轮。首批清单控制在 3 至 5 个词，单轮安全上限维持 50 条；不得为实现逐词 10 分钟而增加浏览器并发或采集频率。
2. **状态阈值**：连续 2 次完整成功采集未出现才标为 `off_shelf`；第一次为 `suspected_missing`，不推首页、保留购物车并提示。后续误下架较多时才评估调为 3 次或增加低频合规详情复查。
3. **部署拓扑**：x-comments 与 shopping 部署在同一台云服务器。Catalog Sync 默认仅绑定宿主机 `127.0.0.1`，shopping 通过回环地址和 Bearer token 同步；不将该端口直接暴露公网。若未来拆为多服务器，必须另行启用 HTTPS、来源限制和网络策略。
4. **数据与密钥运维**：x-comments 部署负责人（或同一运维）负责 PostgreSQL 迁移与每日备份，POC 备份保留 7 天、需要时可扩展至 14 天；同一角色将 `CATALOG_SYNC_TOKEN`、登录态及其他服务端 secret 写入云端密钥管理，不进入 Git 或聊天；采集/同步失败告警先发送至既有运维群或邮箱。
5. **购物与下单边界**：闲鱼商品当前可加入 shopping 购物车，单件数量上限为 1；但**不得**进入结算、扣款、采购或订单流程。shopping 结算页和服务端必须同时拒绝该来源商品；若未来允许交易，必须新建独立 Goal 并重新确认最终价格、库存、卖家可用性和履约责任。
6. **shopping 对接范围**：对接仓库为 `D:\Repo\c-shopping-jp-poc`，使用独立 MongoDB 商品镜像和持久化同步游标；不得直连 x-comments PostgreSQL。

## 10. 明确禁止项

- 为了让每个关键词高频运行而并发开多个浏览器、代理池、账号池或绕过风控；
- 因一次空结果、网络错误、认证失效或同步失败批量标记下架；
- 爬取时长时间锁表/锁库；
- shopping 直连 x-comments PostgreSQL、共享 schema，或使用 x-comments 数据库账号；
- 把 SQLite 作为本阶段运行时数据库、跨服务交接方式或 PostgreSQL 失败时的静默降级；
- 把云端数据库密码、Cookie、Token、登录态、备份文件提交进任一仓库；
- 将闲鱼采集数据描述为实时库存、价格或可履约保证；
- 未完成当前 Goal 的全部验收就开始下一 Goal。

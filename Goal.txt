# 项目任务：闲鱼关键词商品采集闭环 POC

## 1. 项目背景

我们准备开发一个商品采集系统。

用户在一个最小前端页面输入闲鱼搜索关键词，例如：

```text
女生发饰
```

系统创建异步采集任务，通过正常可访问的闲鱼页面获取商品信息，将商品保存到数据库，并通过 REST API 返回给前端展示。

后续项目会扩展为：

* 多关键词持续采集；
* 每5分钟运行一次调度器；
* 增量发现新商品；
* 商品状态复查；
* 已售出或已下架商品自动从公开网站隐藏；
* 保留商品历史数据用于选品分析。

但本次首先完成最小闭环，不得提前实现全部未来功能。

---

# 2. 总体开发原则

采用严格的 Goal Queue 开发方式。

每一个 Goal 都必须包含：

1. 明确目标；
2. 输入和输出；
3. 验收标准；
4. 自动化测试；
5. 人工验收步骤；
6. 失败后的删除或返工规则；
7. 允许继承到下一个 Goal 的成果；
8. 下一个 Goal 的定义。

未经当前 Goal 验收通过，不得进入下一个 Goal。

“部分完成”“理论上可行”“代码已经写好”“静态检查通过”均不能视为验收通过。

真实采集相关 Goal 必须实际运行并取得真实结果，不能仅依靠 Mock、伪造数据或静态 HTML 宣称成功。

---

# 3. 目标队列状态

每个 Goal 的状态只能为：

```text
PENDING
RUNNING
BLOCKED
FAILED
PASSED
```

状态转换规则：

```text
PENDING → RUNNING
RUNNING → PASSED
RUNNING → FAILED
RUNNING → BLOCKED
FAILED → RUNNING
BLOCKED → RUNNING
```

只有 `PASSED` 才能进入下一个 Goal。

每个 Goal 完成后，必须在以下文件记录结果：

```text
docs/goals/goal-N-report.md
```

报告必须包含：

* 本 Goal 的目标；
* 实际完成内容；
* 修改文件；
* 执行命令；
* 测试结果；
* 人工验收结果；
* 未解决问题；
* 是否达到全部验收标准；
* 最终状态；
* 下一 Goal 是否允许启动。

---

# 4. 技术栈

优先采用：

* Python 3.11+
* FastAPI
* Playwright
* SQLAlchemy 2
* Alembic
* Pydantic 2
* SQLite：本地开发
* PostgreSQL：生产环境
* pytest
* Docker Compose

允许根据调研结果提出调整，但不得在没有说明理由的情况下擅自更换整个技术栈。

数据库连接必须通过环境变量配置。

第一版不需要 Redis、Celery、Kafka、WebSocket、代理池或多账号池，除非能够证明没有这些组件就无法完成当前 Goal。

优先采用简单、可运行、可验证的实现。

# 6. 本次 POC 的最终交付目标

最终必须完成以下完整链路：

```text
用户打开最小测试页面
        ↓
输入“女生发饰”
        ↓
点击开始采集
        ↓
前端调用 POST /api/v1/crawl-jobs
        ↓
后端立即返回 job_id
        ↓
后台使用 Playwright 执行真实闲鱼搜索
        ↓
解析商品并写入数据库
        ↓
前端轮询任务状态
        ↓
任务完成后查询商品接口
        ↓
页面展示真实商品标题、价格、图片和链接
```

本阶段不承诺抓取闲鱼全部女生发饰，只要求打通当前正常可访问条件下的关键词商品采集闭环。

# 8. Goal 1：离线后端骨架

## 目标

在不访问真实闲鱼的情况下，完成可测试的后端骨架、数据库模型、任务模型和商品解析接口。

## 建议目录

```text
app/
  api/
  crawler/
  models/
  schemas/
  services/
  repositories/
  jobs/
  core/
tests/
fixtures/
docs/
```

目录可以调整，但必须保证模块职责清晰。

## 商品模型

至少包含：

```text
item_id
title
price
image_url
item_url
location
source
first_seen_at
last_seen_at
created_at
updated_at
```

商品和关键词之间应建立关联关系，不得仅在商品表中保存一个关键词字符串。

## 任务模型

至少包含：

```text
job_id
keyword
status
created_at
started_at
finished_at
discovered_count
new_count
updated_count
duplicate_count
error_count
error_message
```

任务状态至少包括：

```text
pending
running
succeeded
partially_succeeded
failed
```

## API

实现：

```text
GET  /health
POST /api/v1/crawl-jobs
GET  /api/v1/crawl-jobs/{job_id}
GET  /api/v1/items
GET  /api/v1/items/{item_id}
```

创建任务接口必须立即返回，不得阻塞等待爬虫完成。

## 离线测试数据

在 `tests/fixtures/` 中保存经过脱敏的 HTML fixture 或结构化测试数据。

解析器必须能够在不访问真实网站的情况下执行测试。

## 硬验收标准

必须同时满足：

* 数据库迁移成功；
* SQLite 下可以运行；
* PostgreSQL 配置已预留；
* 商品 `item_id` 唯一；
* 重复商品不会重复插入；
* 同一商品可以关联多个关键词；
* 创建任务接口立即返回 `job_id`；
* 商品列表支持分页；
* 商品列表支持关键词过滤；
* 所有测试不依赖真实闲鱼网络请求；
* `pytest` 全部通过；
* `/docs` 可以正常生成；
* `/health` 返回正常；
* 没有使用伪造代码假装执行真实爬虫。

## 失败返工规则

出现以下任意情况，Goal 1 失败：

* API 请求会同步等待爬虫完成；
* 数据库中重复插入同一商品；
* 业务逻辑和页面选择器混在 API 路由中；
* 测试必须访问真实网站；
* 数据库迁移不能从空库执行；
* 为了未来功能提前引入大量复杂组件。

失败后：

* 删除未被当前 Goal 使用的复杂组件；
* 保留通过测试的数据模型和接口；
* 修复不符合项；
* 重新运行全部测试。

## 通过后保留

保留：

* 数据库模型；
* 数据迁移；
* API；
* 数据访问层；
* 任务模型；
* 解析器；
* 离线测试。

通过后进入 Goal 2。

# 9. Goal 2：真实闲鱼关键词采集

## 目标

在人工提供合法登录状态后，真实搜索：

```text
女生发饰
```

并将真实商品保存到数据库。

## 必须实现

1. 使用 Playwright；
2. 支持有头模式完成首次人工登录；
3. 保存浏览器认证状态；
4. 后续任务复用认证状态；
5. 输入关键词；
6. 打开当前正常可访问的闲鱼搜索结果；
7. 按配置抓取有限范围；
8. 提取商品；
9. 标准化字段；
10. 保存数据库；
11. 记录任务结果；
12. 登录失效时安全失败并提示重新登录。

## 第一版抓取范围

默认：

```text
keyword = 女生发饰
max_pages = 3
```

如果当前页面不是页码模式，可以改为：

```text
max_items = 50
```

但必须在文档中说明实际采用的边界。

## 第一版商品字段

真实采集至少获得：

```text
item_id
title
price
image_url
item_url
location（页面可获得时）
keyword
first_seen_at
last_seen_at
```

不得伪造缺失字段。

## 强制安全限制

不得实现：

* 验证码绕过；
* 验证码识别；
* 签名破解；
* 访问控制绕过；
* 登录绕过；
* 高频并发抓取；
* 代理池；
* 多账号轮换；
* 私聊数据采集；
* 手机号采集；
* 精确个人地址采集；
* 非公开个人信息采集。

遇到验证码、登录失效或访问限制时：

```text
安全停止任务
记录失败原因
保留已有数据
不得将任务伪装为成功
```

## 硬验收标准

必须同时满足：

* 使用真实关键词“女生发饰”完成至少一次真实任务；
* 抓到至少20条不同商品，或者抓到当前配置范围内全部真实可见商品；
* 若真实可见商品少于20条，必须提供页面证据和实际数量，不得伪造；
* 商品链接可以打开并对应真实闲鱼商品；
* 商品标题、价格和主图与页面一致；
* 商品 ID 可以稳定去重；
* 连续运行两次不会重复插入同一商品；
* 第二次运行会更新 `last_seen_at`；
* 任务报告包含发现、新增、更新、重复和错误数量；
* 登录失效时任务明确失败；
* 真实执行结果写入：
  `docs/live-test-report.md`；
* 真实结果中抽查至少10条，字段正确率达到100%。

## “抽查正确率100%”定义

人工随机抽查10条商品：

* 标题正确；
* 价格正确；
* 主图对应；
* 链接对应；
* 商品 ID 没有串号。

任何一条不正确，Goal 2 不通过。

## 失败返工规则

出现以下任意情况，Goal 2 失败：

* 使用 Mock 数据冒充真实结果；
* 抓到的数据无法对应真实链接；
* 两次运行产生大量重复记录；
* 页面解析失败但任务仍标记成功；
* 登录失效导致旧商品被错误删除；
* 选择器散落在多个业务文件；
* 真实抓取完全不可复现。

失败后：

* 不删除已经确认正确的数据库和 API 代码；
* 删除错误解析器和无效选择器；
* 保存失败页面或错误上下文；
* 将失败原因写入：
  `docs/debug/`；
* 修复后重新完成真实采集和10条人工抽查。

## 通过后保留

保留：

* Playwright 登录态；
* 搜索流程；
* 商品解析器；
* 去重逻辑；
* 真实采集报告；
* 错误日志；
* 数据库结果。

通过后进入 Goal 3。

# 10. Goal 3：最小前端闭环

## 目标

完成一个仅用于 POC 验收的最小测试页面。

这不是正式产品前端，不要求复杂 UI。

## 页面必须包含

1. 关键词输入框；
2. 默认值：

   ```text
   女生发饰
   ```
3. “开始采集”按钮；
4. 当前任务状态；
5. 任务统计；
6. 错误提示；
7. 商品列表；
8. 分页或加载更多。

商品卡片至少显示：

* 主图；
* 标题；
* 价格；
* 地区；
* 闲鱼原始链接。

## 前端流程

```text
输入关键词
→ 创建任务
→ 获得 job_id
→ 轮询任务状态
→ 任务成功
→ 查询商品列表
→ 展示商品
```

前端不能直接调用闲鱼页面，只能调用本项目后端 API。

## 硬验收标准

必须同时满足：

* 输入“女生发饰”后可以创建任务；
* 创建任务接口在短时间内返回 `job_id`；
* 页面显示 `pending` 或 `running`；
* 任务完成后自动显示商品；
* 页面展示的商品来自数据库；
* 页面商品可点击跳转到真实闲鱼链接；
* 后端失败时页面给出明确错误；
* 页面不会无限转圈；
* 刷新页面后仍能查询历史商品；
* 不需要修改代码即可更换关键词；
* 至少完整演示一次真实端到端流程。

## 失败返工规则

出现以下任意情况，Goal 3 失败：

* 前端展示硬编码商品；
* 前端直接访问闲鱼；
* 点击后页面一直等待，没有任务状态；
* 后端失败但页面仍提示成功；
* 商品列表不是来自数据库；
* 只有 Swagger，没有实际输入页面。

失败后：

* 删除硬编码测试数据；
* 保留可用 API；
* 修复任务状态和前端交互；
* 重新完成完整演示。

## 通过后保留

保留最小测试页面，标记为：

```text
POC internal demo
```

不得将其声称为正式产品前端。

```text
每5分钟调度器运行

选择当前到期的关键词任务
        ↓
检查同关键词任务锁
        ↓
检查登录状态与运行条件
        ↓
抓取规定范围内的搜索结果
        ↓
判断本轮是否完整且可比较
        ↓
本轮完整：
  新商品 → 入库
  已有商品 → 更新
  疑似消失 → 临时隐藏并复查

本轮不完整：
  只新增或更新已看到的商品
  不对未看到商品作状态判断
        ↓
状态复查队列分批访问详情页
        ↓
仍在售 → active + visible
已售出 → sold + hidden
已下架 → removed + hidden
访问异常 → unknown + hidden_pending_check
        ↓
记录任务指标、失败原因和状态历史
```

重要原则：

> 搜索结果未出现只能作为疑似不可用信号，不能直接证明商品已经售出或下架。

# 14. Goal Queue 执行规则

你必须严格按照以下顺序执行：

```text
Goal 0
↓ PASSED
Goal 1
↓ PASSED
Goal 2
↓ PASSED
Goal 3
↓ PASSED
Goal 4
```

每次只执行一个 Goal。

完成当前 Goal 后：

1. 运行测试；
2. 执行人工验收；
3. 生成 Goal 报告；
4. 逐项对照硬验收标准；
5. 将状态设为 `PASSED`、`FAILED` 或 `BLOCKED`；
6. `PASSED` 才能定义并启动下一 Goal；
7. `FAILED` 必须执行删除或返工规则；
8. `BLOCKED` 必须写明阻塞原因和解除条件；
9. 不得为了赶进度降低验收标准；
10. 不得在报告中隐瞒失败项。

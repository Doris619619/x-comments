# 闲鱼商品展示 Demo 接入目标

## Goal 队列与执行规则（必须遵守）

本文件不是可并行勾选的待办清单，而是严格顺序的目标队列。任何时刻只允许一个
Goal 处于执行中；未满足当前 Goal 的全部验收条件时，不得开始下一个 Goal。

### 通用状态与清理约束

| 状态 | 定义 | 允许的后续动作 |
| --- | --- | --- |
| `pending` | 尚未开始，不能修改其目标范围内的代码 | 等待前序 Goal 完成 |
| `in_progress` | 当前唯一允许实施的 Goal | 仅修改本 Goal 所需代码和文档 |
| `passed` | 全部硬性验收已获得实际证据 | 关闭当前 Goal，激活其定义的下一个 Goal |
| `blocked` | 外部状态、权限、真实 API 或需求缺失，无法安全推进 | 记录证据，停止；不伪造数据或降级验收 |
| `failed` | 实现违反边界、回归无法在本 Goal 修复，或两次独立尝试均不能满足硬验收 | 回滚/删除**仅由本 Goal 新增且未被其他功能依赖的临时产物**；保留诊断记录；不自动删除用户已有代码、数据库数据、分支或登录态 |

每个 Goal 的“删除/清理条件”均只描述允许清理的本 Goal 临时文件、失效配置、未引用
组件或未启动的后续队列项；任何 `rm`、数据库清空、Git reset、删除分支或覆盖用户已有
改动，必须在实际执行前单独获得用户确认。

### Goal 0：基线冻结与现有详情页故障修复

**状态：`pending`。这是唯一的起始 Goal。**

**目标定义**

在不改变现有商品业务含义的前提下，修复 `c-shopping-jp-poc` 现有商品详情页白屏，并建立
可重复的基线验证。该 Goal 不接入闲鱼 API、不增加新的商城入口。

**高门槛验收（全部满足才可通过）**

1. 修改前记录两个仓库的 `git status --short`；不得覆盖其中任意用户已有改动。
2. `http://127.0.0.1:3000/products/6a546477598e321a66b9b15d` 可以通过浏览器直接访问，页面包含商品标题、至少一张图片和面包屑或安全降级面包屑；控制台不存在未捕获运行时错误。
3. 通过商城正常商品卡片点击进入同一详情页，URL 发生正确站内导航，页面不白屏。
4. 至少验证一个缺少/不完整 `categoryLevels` 的商品和一个分类完整商品；两者均不因 `Breadcrumb` 抛错。
5. 为 `Breadcrumb` 的不完整分类输入增加自动化测试或可重复运行的专门验证脚本；测试必须证明“无分类、仅一级、三级完整”均不会读取未定义的 `.slug`。
6. `npm run lint` 与 `npm run build` 通过；若项目既有 warning，必须与本次新增 warning 区分记录。
7. 更新商城相关文档，说明详情页容错行为和实际验证结果。

**blocked 条件**

- 无法访问或启动本地 MongoDB/Next.js，且无法获得真实页面与数据；
- 当前已有未提交改动与 `Breadcrumb` 或商品详情页修改发生冲突，无法安全合并；
- 无法获得 `c-shopping-jp-poc` 的写入权限。

**failed 后允许的删除/清理条件**

- 仅删除本 Goal 新增、未被测试引用的临时诊断日志、截图和临时脚本；
- 若新增的容错实现造成其他详情页回归，撤销本 Goal 自己的改动，而不触碰用户原有修改；
- `Goal 1` 仍保持 `pending`，不得以假的静态详情页绕过本 Goal。

**通过后队列动作**

- 标记 Goal 0 为 `passed`；
- 激活且仅激活 Goal 1。

### Goal 1：跨服务商品 API 契约与可用性证明

**状态：`pending`，仅在 Goal 0 通过后激活。**

**目标定义**

确认 `x-comments` 能以真实数据库数据稳定提供商城所需的只读商品 API；补齐最小契约测试与
服务端配置说明。此 Goal 不在商城渲染闲鱼商品。

**高门槛验收（全部满足才可通过）**

1. FastAPI 在本地可启动；`GET /health` 返回数据库健康状态。
2. 使用真实 `x-comments` 数据库中至少一条由既有采集链路写入的商品（不临时造假数据），验证：
   - `GET /api/v1/items?page=1&page_size=...` 返回分页；
   - `GET /api/v1/items/{item_id}` 返回与列表同一商品；
   - 字段包含 `item_id`、`title`、`price`、`image_url`（允许 null）、`location`（允许 null）、`last_seen_at` 与 `source=xianyu`。
3. 无商品、缺图、无地区、未知 item ID（404）、非法分页参数（422）均有稳定 API 行为且由测试覆盖。
4. 商城以服务器端 HTTP 调用为设计前提；不为浏览器调用而宽松开放 CORS。若确需 CORS，来源必须由配置显式限定并有测试。
5. `pytest`、`ruff check .`、`mypy app` 通过（如现有命令不同，记录实际命令和结果）；中文 Docstring 检查通过。
6. 更新 `x-comments` 的 `docs/api.md`、`docs/architecture.md`、`docs/known-limitations.md` 及 `.env.example`（仅在确有配置变化时），描述真实服务端对接方式，绝不记录 Cookie/Token/登录态。

**blocked 条件**

- 真实采集数据不存在，且无法在不伪造数据的前提下提供可访问的真实记录；
- Playwright 登录失效/风控导致采集停止；不得绕过，记录为 `blocked_by_auth_or_risk_control`；
- 服务间地址、端口或网络不可达且无法由安全配置解决。

**failed 后允许的删除/清理条件**

- 删除本 Goal 新增但未接入的临时 API 探测脚本和测试数据库；
- 不删除已有真实商品、迁移、采集任务或 `storage_state.json`；
- Goal 2 保持 `pending`。

**通过后队列动作**

- 标记 Goal 1 为 `passed`；
- 激活且仅激活 Goal 2。

### Goal 2：只读闲鱼 Catalog Provider

**状态：`pending`，仅在 Goal 1 通过后激活。**

**目标定义**

在 `c-shopping-jp-poc` 实现独立的服务器端 `XianyuCatalogProvider`。它把 FastAPI 响应映射为
只读商品视图模型，但不改变现有 MongoDB Provider 的行为，也不接入交易流程。

**高门槛验收（全部满足才可通过）**

1. Provider 通过 `XIANYU_API_BASE_URL` 与明确的 `CATALOG_PROVIDER=xianyu`（或等价显式开关）启用；默认商城数据源仍为 MongoDB。
2. 所有 API 调用只能发生在 Next.js 服务器端；服务地址与任何未来鉴权信息均不得进入 `NEXT_PUBLIC_*`、浏览器 bundle 或客户端日志。
3. 映射准确保留闲鱼 `item_id` 作为稳定外部标识；对象必须包含 `source: 'xianyu'`、`readOnly: true`、`currency: 'CNY'` 与 `item_url`（仅内部字段）。
4. 价格不经 JPY 转换、不调用 `formatJPY`；金额精度不丢失。
5. 图片 URL 只允许在受控 Next.js remote pattern / `unoptimized` 策略下展示；图片缺失或加载失败能回退本地占位图。
6. API 超时、5xx、格式非法、空列表和 404 都被 Provider 显式处理，前台可显示日语失败/空状态，不能抛出未处理异常或白屏。
7. 新增 Provider 的单元/契约测试覆盖列表、详情、空数据、缺图和 API 失败；`npm run commerce:verify`、`npm run lint`、`npm run build` 通过。

**blocked 条件**

- Goal 1 的 API 契约不能稳定满足；
- 因 Next.js 图片安全策略无法展示远程公开图片，且缺少合规的占位降级路径；
- Provider 映射会迫使订单服务信任外部价格/库存，且无法通过只读边界隔离。

**failed 后允许的删除/清理条件**

- 删除未被路由引用的 Provider、环境变量示例与其测试 fixture；
- 恢复 `CATALOG_PROVIDER` 默认值，保留问题诊断；
- 不修改或删除 MongoDB 的商品、订单、用户数据；Goal 3 保持 `pending`。

**通过后队列动作**

- 标记 Goal 2 为 `passed`；
- 激活且仅激活 Goal 3。

## 背景与最终目标

将 `D:\Repo\x-comments`（闲鱼关键词采集 FastAPI 服务）接入
`D:\Repo\c-shopping-jp-poc`（Next.js 日语商城），完成一个仅用于“发现和展示”的 Demo。

用户在商城内可浏览从闲鱼采集的商品列表，并可进入商城内的商品详情页。这个阶段不跳转闲鱼、不实现交易、不接入购物车、订单、支付或聊天。

## 已确认的产品边界

### 要做

1. 商城新增可见的“闲鱼精选”入口或专区，日语文案使用“闲鱼セレクト”或同等自然日语。
2. 从 `x-comments` 的真实 REST API 读取商品：
   - `GET /api/v1/items?page=...&page_size=...&keyword=...`
   - `GET /api/v1/items/{item_id}`
3. 商品列表和站内详情页展示：标题、人民币价格、图片、地区、商品 ID、采集/最近发现时间。
4. 面向用户的新增文案均使用日语。
5. 在商品详情页清晰注明：
   - `閑魚から収集した参考情報です。`
   - `価格・在庫・掲載状況は更新される場合があります。`
6. 当 API 暂无商品、图片缺失或 API 不可用时，提供日语空状态/失败状态，不能白屏。
7. 修复现有商城商品详情页白屏问题（见“已复现问题”）。
8. 为本次新行为更新商城项目中的相应文档；为 x-comments 的跨服务公开访问配置同步更新 `docs/`。

### 明确不做

- 不跳转闲鱼原商品链接；`item_url` 只可作为内部数据，不在页面暴露为跳转入口。
- 不显示“加入购物车”“立即购买”“配送”“退货”“库存保证”等交易或自营承诺。
- 不接入订单、支付、卖家沟通、库存锁定、价格换算或汇率服务。
- 不复制、下载或二次托管闲鱼图片；Demo 先直接使用采集到的公开 `image_url`，缺图使用商城本地占位图。
- 不实现验证码绕过、登录绕过、签名破解、代理池、自动重试风控拦截等任何规避平台控制的功能。

## 推荐技术方案

保持两个仓库、两个进程和各自数据库独立，用 HTTP API 对接；不要把 Python/FastAPI 代码并入 Next.js，也不要让商城直接读取 x-comments 的数据库。

```text
x-comments (FastAPI + SQLite/PostgreSQL + Playwright)
  └─ /api/v1/items
        ↓ HTTP（服务端到服务端）
c-shopping-jp-poc (Next.js)
  └─ XianyuCatalogProvider
        ↓
闲鱼精选列表与站内详情页
```

### 建议的商城实现

1. 新建独立数据源，不破坏现有 `MongoDbCatalogProvider`：
   - 例如 `lib/commerce/xianyu-catalog-provider.js`；
   - 配置项建议为 `CATALOG_PROVIDER=xianyu` 和服务端变量 `XIANYU_API_BASE_URL`；
   - 仅在服务器端调用 FastAPI，绝不能把服务端地址或后续凭据放进 `NEXT_PUBLIC_*`。
2. 不要强行复用现有“可交易商品”的所有字段。可新增只读映射/页面组件，或给 Provider 映射添加明确的 `source: 'xianyu'`、`readOnly: true`、`currency: 'CNY'`。
3. 闲鱼商品详情页应是独立只读视图：不渲染 `AddToCart`、`FreeShipping`、`Services`、库存/尺码/颜色选择、评论提交等组件。
4. 价格原样以 CNY 显示，例如 `￥12.50（人民元）`；不要调用目前只支持 JPY 的 `formatJPY`。
5. `image_url` 需要纳入 `next.config.js` 的图片 remote pattern，或者对这组远程公开图片明确采用 `unoptimized`。图片加载失败必须降级为 `/placeholder.png`。
6. 在商城首页增加“闲鱼精选”模块或明确导航入口；先做成可开关的 Demo，避免取代当前 MongoDB 商品商城。

### 建议的 x-comments 配置与服务端改动

1. 现有 FastAPI 已提供物品 API，字段为：
   - `item_id`, `title`, `price`, `image_url`, `item_url`, `location`, `source`,
     `first_seen_at`, `last_seen_at`, `created_at`, `updated_at`。
2. 若 Next.js 仅服务端请求 FastAPI，通常不需要 CORS；若增加浏览器端请求，必须通过显式、可配置的允许源实现 CORS，不能使用通配符或暴露敏感端点。
3. 从 `.env.example` 增加无敏感信息的服务地址示例和运行说明；不得提交 `storage_state.json`、Cookie、Token 或登录态。

## 已复现问题：点击商品详情后白屏

### 复现

在 `http://127.0.0.1:3000/` 点击商品卡片，例如：

```text
/products/6a546477598e321a66b9b15d
```

直接访问详情 URL 后，浏览器控制台出现：

```text
TypeError: Cannot read properties of undefined (reading 'slug')
at Breadcrumb (components/product/Breadcrumb.jsx:48:64)
```

根因：`components/product/Breadcrumb.jsx` 假设 `categoryLevels` 的所有层级均存在；当前种子商品缺少其中某级，组件直接读取未定义对象的 `.slug`，导致整个 React 详情页错误边界渲染为空。

### 修复要求

1. `Breadcrumb` 必须过滤缺失层级并安全渲染；没有有效分类时仅显示首页/商品，不能抛异常。
2. 增加一个覆盖“categoryLevels 不完整”的自动化测试，或至少写入可重复运行的验证脚本。
3. 修复后验证该 URL 和其他种子商品详情均不白屏。

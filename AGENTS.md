# AGENTS.md

## 一、项目介绍

本项目是一个“闲鱼商品关键词采集后端 POC”。

当前目标是打通以下最小闭环：

```text
用户在前端输入关键词“女生发饰”
        ↓
前端创建采集任务
        ↓
后端使用 Playwright 访问闲鱼搜索页面
        ↓
抓取当前正常可访问的真实商品
        ↓
清洗、去重并保存到数据库
        ↓
通过 REST API 返回给前端展示
```

当前阶段只验证“关键词输入 → 真实采集 → 数据入库 → API 返回 → 前端展示”这一条完整链路。

本项目当前不承诺：

- 抓取闲鱼全部商品；
- 覆盖全部女生发饰；
- 秒级实时更新；
- 自动判断所有商品是否售出；
- 绕过验证码、登录限制或平台风控。

后续可能扩展：

- 多关键词采集；
- 每 5 分钟运行一次调度器；
- 增量发现新商品；
- 商品价格和状态更新；
- 疑似下架商品复查；
- 已售出或已下架商品自动从公开网站隐藏；
- 保留历史数据用于后续选品分析。

当前开发必须优先完成最小 POC，不得为了未来功能过度设计。

## 二、当前核心交付目标

当前阶段必须实现：

1. 前端可以输入“女生发饰”；
2. 后端创建异步采集任务并返回任务 ID；
3. 使用 Playwright 进行真实闲鱼关键词搜索；
4. 将商品标题、价格、图片、链接等字段保存到数据库；
5. 使用商品唯一 ID 去重；
6. 前端可以查询任务状态；
7. 前端可以分页展示数据库中的真实商品；
8. 登录失效或出现风控时安全停止；
9. 不实现任何验证码绕过、签名破解或访问控制绕过功能。

当前阶段不应提前引入：

- Redis；
- Celery；
- Kafka；
- 多账号池；
- 代理池；
- WebSocket；
- 分布式任务系统；
- AI 选品；
- 自动购买；
- 自动联系卖家。

除非当前目标确实无法完成，否则优先使用简单、可维护、可测试的方案。

## 四、中文注释要求

### 1. 文件级注释

所有可添加注释的源代码文件，开头必须使用中文说明：

- 这个文件负责什么；
- 它属于哪个模块；
- 它与其他模块的关系；
- 它不应该负责什么。

示例：

```python
"""
本文件负责闲鱼关键词采集任务的调度与状态管理。

职责：
1. 创建和执行采集任务；
2. 更新任务状态；
3. 调用爬虫服务和数据存储服务。

本文件不负责页面解析和数据库底层操作。
"""
```

对于 JSON 等不支持注释的文件，应在相邻的 `docs/` 文档中说明其用途。

### 2. 函数级注释

每个函数、方法和类前必须有中文注释或中文 Docstring，说明：

- 该函数做什么；
- 主要输入是什么；
- 返回什么；
- 可能抛出什么异常；
- 是否有副作用。

示例：

```python
def create_crawl_job(keyword: str) -> CrawlJob:
    """
    创建一个新的关键词采集任务。

    参数：
        keyword: 用户输入的闲鱼搜索关键词。

    返回：
        新创建的采集任务对象。

    异常：
        ValueError: 关键词为空或不合法时抛出。
    """
```

禁止只写“初始化”“处理数据”“执行任务”这类没有信息量的注释。

## 五、模块拆分要求

必须遵守单一职责原则。

不要把以下逻辑放在同一个文件中：

- API 路由；
- 业务逻辑；
- Playwright 页面操作；
- 页面解析；
- 数据库读写；
- 任务调度；
- 配置读取；
- 日志与异常处理。

推荐分层：

```text
app/
  api/             # API 路由
  crawler/         # Playwright 页面访问与解析
  services/        # 业务逻辑
  repositories/    # 数据库访问
  models/          # 数据库模型
  schemas/         # 请求和响应模型
  jobs/            # 任务调度与后台任务
  core/            # 配置、日志、异常、公共能力
```

当一个文件出现以下情况时，应优先拆分：

- 同时承担多个职责；
- 代码明显难以阅读；
- 修改一个功能容易影响无关功能；
- 多个函数重复使用同一组内部逻辑；
- 文件持续膨胀；
- 测试难以独立编写。

原则上：

- 单个函数尽量只做一件事；
- 避免超长函数；
- 避免超大类；
- 避免循环依赖；
- 公共逻辑应下沉到独立模块；
- 页面选择器必须集中管理，不得散落在业务代码中。

## 六、可维护性要求

开发时必须考虑：

- 明确的数据流；
- 稳定的模块接口；
- 合理的异常边界；
- 可替换的实现；
- 可测试的依赖；
- 明确的类型标注；
- 可追踪的日志；
- 不重复造轮子；
- 不复制粘贴相同逻辑。

关键要求：

1. 业务逻辑不得直接依赖具体数据库实现；
2. API 路由不得包含复杂业务逻辑；
3. 爬虫解析器应支持离线 fixture 测试；
4. 配置项不得硬编码；
5. 敏感信息不得写入代码或日志；
6. 失败必须有明确异常和状态，不得静默吞错；
7. 所有重要状态变化必须可追踪；
8. 新增功能时必须考虑回归影响。
